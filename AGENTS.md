# AGENTS.md

Operational guide for AI coding agents working in this repo. End-user
documentation lives in [`README.md`](README.md); this file only covers
things that aren't obvious from the code itself.

## Stack

- Python `>=3.10.12`, managed with [`uv`](https://docs.astral.sh/uv/).
- CLI built with [`cyclopts`](https://cyclopts.readthedocs.io/), TOML
  manipulation with [`tomlkit`](https://github.com/python-poetry/tomlkit),
  formatting via `ruff`, and `taplo` / `tombi` for `pyproject.toml`.
- Logging is `lfp_logging` - never `logging.getLogger`, never `print`.

## Quick reference

```bash
uv sync                                 # set up the dev environment
uv run pytest                           # run the full test suite
uv run ruff format --check .            # verify formatting
uv run ruff format .                    # apply formatting
uv run ruff check src/lfp_build/...     # lint a touched file (scope it)
uv run lfp-build sync                   # exercise the tool against this repo
uv run lfp-build readme update          # refresh README sentinel blocks
mise run publish                        # ruff-fix + commit + bump + push tags
```

The full pytest suite must pass with no `--deselect` flags. `temp_workspace`
in [`tests/conftest.py`](tests/conftest.py) provides a git-initialized uv
workspace named `test-workspace`; member dirs created via
`lfp-build add` are root-prefixed by default
(`add foo` -> `packages/test-workspace-foo/`).

## Project layout

See README.md `## Module Reference` for the full per-file breakdown. In
short:

- `src/lfp_build/cli.py` - cyclopts entry point, mounts each verb.
- `src/lfp_build/commands/<verb>.py` - one file per CLI verb.
- `src/lfp_build/{workspace,pyproject,version,names,util,_config,_hooks (merged into commands/hooks.py)}.py` - shared primitives.
- `src/lfp_build/templates/` - bundled resource files (loaded via
 `importlib.resources`, included in wheel + sdist).
- `src/lfp_build/docs/` - **tracked** agent-facing bundle
 (`<name>/SKILL.md` for skills, `<name>.md` for reference docs) that
 ships in the wheel and powers `lfp-build skills install` /
 `lfp-build docs install`. This directory is the single source of
 truth: authored, committed, and packaged from the same location so
 PyPI wheels, `pip install git+...` clones, and editable installs all
 carry the bundle without a separate staging step.
- `tests/` - flat `def test_*` functions, no test classes.

The package layout intentionally mirrors the CLI tree: when adding a new
verb, create `commands/<verb>.py` and mount it in `cli.py`.

## Bundle contents (self-hosting the SKILL / docs)

The SKILL.md at `src/lfp_build/docs/lfp-build-workspace/SKILL.md` and the
reference guide at `src/lfp_build/docs/lfp-build-conventions.md` are
exactly what `lfp-build skills install` / `lfp-build docs install` ship
to consumer projects. When working on this repo, follow them as though
they were installed here - they document the workspace-member layout,
the sync verbs, and the coding conventions that this repo also expects
of contributors. Edits to those files apply immediately at runtime (the
bundle is read via `importlib.resources` from the same on-disk path) and
will land in the next published wheel.

## Publish workflow

`mise.toml` exposes two tasks:

- `mise run ruff-fix` - runs `uv run ruff check . --fix --unsafe-fixes`
  followed by `uv run ruff format .`.
- `mise run publish [patch|minor|major]` - depends on `ruff-fix`, then
  `git add .`, commits any staged changes as `pre-release: staging
  changes`, invokes `uv run bump-my-version bump <part>`, and finally
  pushes `HEAD --tags` to `origin`. Never run this without explicit user
  consent (see "What never to do without explicit consent" below).

CI/CD builds the wheel from the tracked tree, so no separate staging
step exists (or is needed): `uv_build` picks up everything under
`src/lfp_build/docs/` as package data via the `lfp_build.docs`
subpackage.

## README sentinels

Help output for every CLI verb is embedded in `README.md` between
`<!-- BEGIN:cmd lfp-build <verb> --help -->` / `<!-- END:cmd -->` markers
and is regenerated by:

```bash
uv run lfp-build readme update --filter '<verb> --help'
```

After changing any CLI parameter docstring, default value, or flag name,
regenerate the matching sentinel. The `tests/test_cli_readme.py` suite
will fail loudly if you forget.

## Conventions enforced by the user

The user has personal coding rules that supersede generic best-practice
defaults. Follow them when editing existing code or writing new code:

- **No em / en dashes** (`-` / `--`). Hyphens (`-`) are fine.
- **No emojis** unless explicitly requested.
- **Don't change variable names or method-internal logic** unless asked.
  This applies even when refactoring would be cleaner; if you spot
  something worth renaming, mention it in chat instead of doing it
  silently.
- **Add docstrings to all new code**, and update existing docstrings
  whenever variables or contracts change. Module docstrings live
  *after imports* and before any constants / functions / classes.
- **Standard Python ordering inside a module**: imports, module
  docstring, constants, public functions, private (`_`-prefixed)
  functions, classes, private classes.
- **DRY** - if logic is reused twice, extract to a private helper or to
  `util.py` / `pyproject.py` / `workspace.py` / `names.py` as
  appropriate.
- **Empty `__init__.py`** unless explicitly told otherwise.
- **Avoid `print`**. Use `LOG = logs.logger(__name__)` at module scope
  and log instead.
- **Comments explain *why*, not *what*.** Don't narrate the change you
  just made (e.g. no "// changed from int to float").

## Known gotchas

- **`tomlkit.OutOfOrderTableProxy`**: when a TOML file already has
  several `[tool.X]` sub-tables, accessing `data["tool"]` returns an
  `OutOfOrderTableProxy`, not a real `Table`. The proxy implements
  `MutableMapping` but **not** the `.add()` / `.remove()` methods that
  `tomlkit.Table` exposes. Use `setdefault` / `__setitem__` / `__delitem__`
  to mutate it.
- **`PyProject.table(*keys, create=True)`** is safe on
  `OutOfOrderTableProxy` parents - it walks with `__setitem__`, not
  `.add()` (see `test_table_create_on_out_of_order_table`). Prefer it
  over hand-rolled `setdefault(...)` chains so table creation stays
  consistent across sync steps.
- **`_config.MEMBER_PROJECT_DIRECT_REFERENCE` defaults to `False`** when
  `LFP_BUILD_MEMBER_PROJECT_DIRECT_REFERENCE` is unset. Internal
  workspace deps are emitted as plain names by default; only `=true` /
  `=1` / `=yes` / `=on` flips them to
  `name @ file://${PROJECT_ROOT}/...` form.
- **Top-level workspace whitespace**: `_reorder_document` in
  `pyproject.py` is the sole authority on inter-group blank lines at
  the top level. Trailing `Whitespace` items live at the end of the
  *previous* table's body; trim them via `_strip_top_level_whitespace`
  before inserting `tomlkit.nl()` separators.

## What never to do without explicit consent

- Auto-commit, auto-push, or auto-tag. Stop after lint + tests pass.
- Run any deploy / publish command (`databricks bundle deploy`,
  `databricks apps deploy`, `git push`, `uv publish`, etc.). Local
  builds and validations are fine.
- Edit `git config`, force-push, or skip git hooks (`--no-verify`, etc.).
- Replace `dependencies = ["foo"]` style internal references with
  `dependencies = ["foo>=..."]` - this repo treats workspace members as
  bare names so `[tool.uv.sources]` controls resolution.
- Try / except around an import to make a dep "optional". Either the
  import is available or it isn't; don't paper over it.

## Verifying a change

The minimum bar before handing back to the user:

1. `uv run pytest` passes (currently 61 tests, no deselects).
2. `uv run ruff format --check .` is clean (run `uv run ruff format .`
   if not).
3. `uv run ruff check <touched files>` is clean. The repo has one
   pre-existing `F401` warning on `Collection` in `util.py` that is not
   yours to fix unless you introduce a fresh import that supersedes it.
4. README sentinel blocks for any touched CLI verbs were regenerated.

If a pre-existing failure blocks verification, surface it in chat
rather than papering over it with `--deselect`.
